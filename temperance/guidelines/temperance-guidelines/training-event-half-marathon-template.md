# Event Overlay Template — Half Marathon

Status: reusable overlay / draft.
Profile type: event.
Profile id: half-marathon-template.

## Objective and main limiter

- Convert aerobic fitness into half-marathon-specific endurance and pace support.
- The limiting question is often less about maximum long-duration durability than in a marathon build, and more about how much threshold / specific work can be absorbed coherently.

## Primary specificity axis

- Running load usually remains the primary specific load.
- `Specific hard` should usually center on threshold-to-race-pace durability rather than extreme long-duration work.
- The long run remains useful, but it is not automatically as central as in a marathon build.

## Hard-session priority order

- threshold hard
- specific hard
- long-duration hard
- sharp hard

## Spacing / density preferences

- Protect quality density from becoming sloppy.
- Keep enough spacing that race-specific work remains high quality.
- Do not borrow marathon long-run centrality unless the build truly needs it.

## Progression priorities

- Progress running load and threshold / race-pace durability coherently.
- Keep long-duration work sufficient, but not oversized just because marathon logic exists elsewhere in the doctrine set.

## Alert sensitivity / default responses

- Watch for threshold density, hidden moderate drift, and forcing marathon-style long-run cost into a half-marathon block.
- Default response: hold quality density first, then trim the least event-relevant stress.

## Explicit non-goals

- Do not inherit marathon-specific long-run centrality by default.
- Do not assume sharpness-heavy planning is required.

## Template note

- This is intentionally a scaffold, not a fully authored HM doctrine.
