# Event Overlay Template — Ironman

Status: reusable overlay / draft.
Profile type: event.
Profile id: ironman-template.

## Objective and main limiter

- Build durable event-wide readiness across swim, bike, and run without letting one modality silently break the whole block.
- The main limiter is often multi-factor: total fatigue management, long-duration fueling, and the interaction between bike load and run durability.

## Primary specificity axis

- The build should declare one primary specific control axis for the current block, but it should also map the other event-critical anchors explicitly.
- In many Ironman blocks, support modality is no longer a simple synonym for x-train, because bike and swim are themselves primary event modalities.
- `Specific hard` may include long bike work, bricks, race-pace durability, or other event-relevant combinations.

## Hard-session priority order

- event-specific hard
- long-duration hard
- threshold hard
- sharp hard

## Spacing / density preferences

- Review density across the whole event mix, not only through running.
- Protect the run from hidden carryover cost created by bike density.
- Do not assume the running doctrine's support-modality logic transfers unchanged.

## Progression priorities

- Progress the event-critical modality mix coherently rather than maximizing one modality in isolation.
- Use explicit anchor mapping so long bike, long run, and key brick demands are visible instead of buried inside total load.

## Alert sensitivity / default responses

- Watch for bike-to-run carryover, stacked long-duration stress, and total-load patterns that hide run breakdown.
- Default response: reduce the most expensive non-essential stress first and restore event-balance before re-progressing.

## Explicit non-goals

- Do not import pure running logic unchanged.
- Do not treat swim or bike automatically as support-only modalities.

## Template note

- This is intentionally a scaffold, not a fully authored Ironman doctrine.
